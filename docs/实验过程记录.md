## 7.2 协议设计与程序总体设计

- [x] 组内确定协议类型（不搭载 ACK 的 GBN / 搭载 ACK 的 GBN / 选择重传之一或多种）
- [x] 定义帧字段（`include/datalink.h`）：`kind`、`seq`、`ack`、载荷与 CRC 边界约定
- [x] 确定滑动窗口大小、序号空间、ACK 语义（累积确认或捎带等）
- [x] 估算：重传定时器时限、ACK 定时器时限（指导书 6、11.3 要求定量分析，此处先记初值与迭代）

**设计要点记录：**

---
**日期：** 2026 年 5 月
**参与人：** 张恒基、尹浩铭、林旭东、赵博宇（全员）

1. **协议类型选定**：搭载 ACK 的 Go-Back-N（GBN）。
2. **帧结构定义**（`include/datalink.h` 中 `struct frame`）：
   - `kind`（1 字节）：`FRAME_DATA=1`、`FRAME_ACK=2`（NAK 预留 `FRAME_NAK=3`）。
   - `seq`（1 字节）：数据帧发送序号，模 256（`MAX_SEQ=255`）。
   - `ack`（1 字节）：累积确认序号，语义为「期望接收的下一帧序号」。
   - `data[256]`：网络层分组载荷。
   - CRC-32（4 字节）：帧尾追加，不在结构体内定义。
3. **窗口参数**：
   - `MAX_SEQ=255`，序号空间 256（0~255）。早期 `MAX_SEQ=7` 在默认误码长跑中出现过旧重传帧滞留、序号过快回绕后导致网络层坏分组的问题；定稿版扩展序号空间避免旧帧跨周期误收。
   - `WINDOW_SIZE=5`：在 BDP≈3 基础上放大，缓解全双工对称死锁（原取 3 时洪水利用率约 70%）。
4. **定时器参数**：
   - `DATA_TIMEOUT_MS=600`：略大于 RTT（540 ms），考虑 `phl_sq_len` 排队项后实际超时时间更长。
   - `ACK_TIMEOUT_MS=50`：缩短纯 ACK 空等，尽快释放对端发送窗口。
5. **分工**：张恒基（`datalink.c` / 组长调度）；尹浩铭（`datalink_recv.c`）；林旭东（表 3 统稿与报告 PDF）；赵博宇（构建、`docs/实验过程记录.md`、部分表 3 实测）。
6. **尹浩铭负责（接收侧）**：
   - `datalink_recv.c`：实现接收处理函数 `validate_and_process_frame` 和纯 ACK 发送 `send_pure_ack`。
   - `datalink.h`：声明接口函数与导出 `frame_expected` 的读取方法。

## 7.3 编码与调试

- [x] 在 `src/datalink.c` 实现 `wait_for_event` 主循环与各事件分支
- [x] 实现 `send_frame` / `recv_frame` 与 `crc32` 的配合（指导书 8.9）
- [x] 实现与网络层的 `enable` / `disable` 及与物理层 `PHYSICAL_LAYER_READY` 的配合（8.6、8.8）
- [x] 定时器：`start_timer` / `stop_timer`、`start_ack_timer` / `stop_ack_timer`（8.10）

**主要 Bug 与解决：**

**参与人：** 尹浩铭、张恒基

| 现象 | 原因 | 修复 |
|------|------|------|
| 编译时 `struct frame` 大小与发送长度不一致 | `data[256]` 后若定义 CRC 字段，`sizeof(struct frame)` 大于 3+256，`recv_frame` 缓冲不够 | CRC 不在结构体内定义，发送/接收时在缓冲区末尾手动追加/校验 4 字节 CRC |
| `recv_frame` 首帧段错误 | `recv_frame` 缓冲区过小（256 字节），但完整帧含 CRC 共 263 字节 | 将 `rxbuf` 扩大为 512 字节，`MAX_FRAME_BYTES` 定义为 `FRAME_HDR_LEN + PKT_LEN + 4` |
| `crc32(...) != 0` 校验总是失败 | `recv_frame` 长度参数未包含末尾 4 字节 CRC | 修复 `recv_frame` 传入的缓冲区长度，确保包含 CRC 域的完整帧被读出 |
| 纯 ACK 帧 CRC 校验错误 | 纯 ACK 帧未计算 CRC 就直接 `send_frame` | 在 `send_pure_ack` 中补充 `crc32(buf, 3)` 计算并将 CRC 追加到帧尾 |
| 收到重复 DATA 帧后仍然调用了 `put_packet` | `validate_and_process_frame` 未在失序路径禁止交付 | 增加 `return 2` 分支，失序帧不写入 `data_out`，发送端继续发送重复 ACK |
| 长时间运行后双站利用率归零 | `DATA_TIMEOUT` 重传后未重新 `start_timer`，若重传帧再损坏则无法再次超时 | 在 `resend_window` 和 `update_ack_received` 中保证每次有效操作后重新 `start_timer` |
| 默认误码场景约 40 秒出现 `Network Layer received a bad packet` | `MAX_SEQ=7` 时序号空间过小，旧重传帧可能在物理层队列中滞留到接收端序号绕回后被误收 | 将 `seq/ack` 的 1 字节空间完整用于编号，`MAX_SEQ=255`、`NR_BUFS=256`；重新跑表 3 五场景均无坏分组 |
| 双端洪水利用率约 70% 停滞 | 窗口过小 + 未统一 `phl_ready` 闸门，对称死锁 | `WINDOW_SIZE=5`、`ACK_TIMEOUT_MS=50`，循环末尾 `refresh_network_layer_gate` |

（指导书表 3 若多于 5 行，请按 PDF 补全。）

**结论要点：**

---
**参与人：** 尹浩铭

1. **接收侧验证**：`validate_and_process_frame` 在双端 `--flood --utopia` 场景下运行 15 分钟，CRC 校验正确，无 `bad packet` 中止。`dbg_event` 日志显示 CRC 错误帧约 2% 比例（默认 $P_b = 10^(-5)$），全被正确丢弃。
2. **纯 ACK 验证**：在高误码场景（`-b 1e-4`）下，ACK_TIMEOUT 分支按预期发送纯 ACK 帧，避免窗口死锁；日志可见纯 ACK 发送记录。
3. **帧长度魔数替换**：代码中所有硬编码的 `3`（帧首部长度）和 `4`（CRC 长度）已替换为宏 `FRAME_HDR_LEN` 和 `sizeof(unsigned int)`，提高可维护性。
4. **表 3 完成**：五场景均按 `-t 600` 跑满 10 分钟，日志文件为 `table3-*.log`；所有场景自然 `Quit`，无 `bad packet` / `Abort`。
